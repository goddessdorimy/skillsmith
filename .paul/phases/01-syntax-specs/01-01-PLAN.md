<objective>
## Goal
Create concrete syntax spec files for the 3 core folder types: entry point, tasks, and templates. Each spec defines the exact structure, required sections, naming conventions, placeholder conventions, and includes good/bad examples grounded from PAUL `src/` patterns.

## Purpose
These specs are the core IP of the framework — they define what makes a skill file compliant. Without concrete specs, the meta-skill has nothing to enforce and skills remain ad hoc.

## Output
- `specs/entry-point.md` — Syntax spec for `{skill-name}.md` entry points
- `specs/tasks.md` — Syntax spec for `tasks/*.md` workflow files
- `specs/templates.md` — Syntax spec for `templates/*.md` fill-in-the-blank structures
</objective>

<task type="auto">
  <name>Task 1: Create entry point syntax spec</name>
  <files>specs/entry-point.md</files>
  <action>
    Create `specs/entry-point.md` defining the syntax for skill entry points (`{skill-name}.md`).

    Adapt from PAUL `src/rules/commands.md` patterns:
    - YAML frontmatter: name, type (suite|standalone|task-only), version, category, description
    - Add `allowed-tools` field (from PAUL commands)
    - XML sections in order: `<activation>`, `<persona>`, `<commands>`, `<routing>`, `<greeting>`
    - Core principle: Entry points are thin — route to tasks/ for heavy work, load frameworks/ for knowledge

    Structure the spec file with:
    1. Header with purpose description
    2. YAML frontmatter specification (table of fields with required/optional, purpose, examples)
    3. Section order specification (each XML section: what it contains, when to use)
    4. Naming conventions (kebab-case directory, matching filename)
    5. Versioning convention (semver in frontmatter, when to bump)
    6. Good vs bad examples (adapted from PAUL command examples)
    7. Anti-patterns section

    Key adaptations from PAUL commands to skill entry points:
    - PAUL `<objective>` → Skill `<activation>` (when/how the skill triggers)
    - PAUL `<execution_context>` → Skill `<routing>` (what tasks/frameworks to load)
    - PAUL `<process>` → Removed (entry points don't contain process, they route to tasks)
    - Add `<persona>` (skills have personas, PAUL commands don't)
    - Add `<commands>` (list of available sub-commands)
    - Add `<greeting>` (initial response when skill loads)

    Avoid:
    - Copying PAUL conventions verbatim without adapting for skill context
    - Using generic XML tags (section, item, block)
    - Making the spec too abstract — include concrete fill-in examples
  </action>
  <verify>Read the file. Confirm: YAML frontmatter table exists with all fields, all XML sections documented, good/bad examples present, anti-patterns section exists, version field documented</verify>
  <done>AC-1 satisfied: Entry point spec has frontmatter fields, XML sections, naming, versioning, examples, anti-patterns</done>
</task>

<task type="auto">
  <name>Task 2: Create tasks syntax spec</name>
  <files>specs/tasks.md</files>
  <action>
    Create `specs/tasks.md` defining the syntax for task files (`tasks/*.md`).

    Adapt from PAUL `src/rules/workflows.md` patterns:
    - No YAML frontmatter (same as PAUL workflows)
    - Required XML sections: `<purpose>`, `<user-story>`, `<steps>`, `<output>`, `<acceptance-criteria>`
    - Step elements: `<step name="snake_case">` with optional `priority` attribute
    - Content inside steps: imperative instructions, numbered sub-steps

    Key adaptations from PAUL workflows to skill tasks:
    - PAUL `<when_to_use>` → Keep (helps skill know when to invoke this task)
    - PAUL `<loop_context>` → Remove (skills don't have PLAN/APPLY/UNIFY loop)
    - PAUL `<required_reading>` → Keep as `<context>` (files to read before starting)
    - PAUL `<references>` → Keep (lazy-load frameworks when needed)
    - Add `<user-story>` section: "As a [role], I want [action], so that [outcome]"
    - Add `<acceptance-criteria>` as plain checklists (NOT BDD — per project decision)
    - Add `<output>` section defining what the task produces

    Structure the spec file with:
    1. Header with purpose
    2. Section order specification
    3. Step element spec (name attribute, priority, content format)
    4. User story format
    5. Acceptance criteria format (plain checklists, not BDD)
    6. Conditional logic patterns (from PAUL's `<if>` elements)
    7. Good vs bad examples
    8. Anti-patterns

    Avoid:
    - BDD Given/When/Then in acceptance criteria (decision: skills use plain checklists)
    - Making tasks too thin (tasks contain the heavy logic, unlike entry points)
  </action>
  <verify>Read the file. Confirm: no frontmatter, all XML sections documented, step element spec present, user-story format shown, acceptance criteria uses plain checklists (not BDD), good/bad examples present</verify>
  <done>AC-2 satisfied: Tasks spec has sections, step attributes, user story format, plain checklist acceptance criteria, examples</done>
</task>

<task type="auto">
  <name>Task 3: Create templates syntax spec</name>
  <files>specs/templates.md</files>
  <action>
    Create `specs/templates.md` defining the syntax for template files (`templates/*.md`).

    Adapt directly from PAUL `src/rules/templates.md` patterns:
    - Header: `# [Name] Template`
    - Intro: Brief description and file naming pattern
    - Fenced code block containing the actual template
    - Placeholder conventions: `[square brackets]` for human-fillable, `{curly braces}` for variable interpolation
    - Field documentation tables below the fenced block
    - Section specifications explaining what goes where

    Structure the spec file with:
    1. Header with purpose
    2. File structure specification (header → intro → fenced block → field docs → section specs)
    3. Placeholder conventions (detailed, with examples)
    4. YAML frontmatter in template content (explaining it's content TO BE GENERATED)
    5. Field documentation table format
    6. Section specification format
    7. Good vs bad examples
    8. Anti-patterns

    This is the most direct adaptation from PAUL — template syntax translates almost 1:1.

    Avoid:
    - Confusing template frontmatter (metadata about the template) with content frontmatter (what gets generated)
    - Over-specifying — templates should be flexible enough for different skill domains
  </action>
  <verify>Read the file. Confirm: fenced block pattern documented, both placeholder types explained with examples, field documentation table format shown, content vs template frontmatter distinction clear, good/bad examples present</verify>
  <done>AC-3 satisfied: Templates spec has structure, placeholders, field docs, frontmatter distinction, examples</done>
</task>
