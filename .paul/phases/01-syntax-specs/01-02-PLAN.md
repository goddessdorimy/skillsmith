<objective>
## Goal
Create concrete syntax spec files for the 4 auxiliary folder types: frameworks, context, checklists, and rules. Each spec follows the same structure established in Plan 01-01 (purpose, section order, naming, examples, anti-patterns).

## Purpose
Completes the full set of 7 syntax specs. Without these, Skillsmith can't scaffold auxiliary folders and skill authors have no standard for supporting files.

## Output
- `specs/frameworks.md` — Syntax spec for read-only reference knowledge
- `specs/context.md` — Syntax spec for business/user context files
- `specs/checklists.md` — Syntax spec for QA validation checklists
- `specs/rules.md` — Syntax spec for authoring rules (meta-skill only)
</objective>

<task type="auto">
  <name>Task 1: Create frameworks syntax spec</name>
  <files>specs/frameworks.md</files>
  <action>
    Create `specs/frameworks.md` defining syntax for framework files (`frameworks/*.md`).

    Adapt from PAUL `src/rules/references.md` patterns:
    - No YAML frontmatter (same as PAUL references)
    - Optional outer XML container matching the concept (e.g., `<value_equation>`)
    - Teaching-oriented structure: purpose, concepts, examples, anti-patterns
    - Read-only — never modified during skill execution
    - Lazy-loaded by tasks when relevant

    Structure the spec with:
    1. Header with purpose
    2. File structure (outer container, sections)
    3. Teaching patterns (explain by contrast, explain WHY)
    4. Loading pattern (@-references from tasks)
    5. Distinction: frameworks vs context vs templates
    6. Good vs bad examples
    7. Anti-patterns

    Key adaptation from PAUL references:
    - PAUL references explain dev concepts → Skill frameworks explain domain knowledge
    - Same structure works: purpose → concepts → examples → anti-patterns
    - Self-contained principle: a task can load one framework without needing others
  </action>
  <verify>Read file. Confirm: no frontmatter, outer XML container documented, teaching patterns shown, loading pattern with @-references, distinction from context/templates, good/bad examples, anti-patterns</verify>
  <done>AC-1 satisfied: Frameworks spec covers structure, teaching patterns, loading, read-only convention</done>
</task>

    No direct PAUL equivalent — context is skill-specific. Define from first principles:
    - Plain markdown with standard sections
    - Two subtypes: user profiles and business profiles
    - May include onboarding templates (questions to gather context)

    Structure the spec with:
    1. Header with purpose
    2. Context subtypes (user profile, business profile, custom)
    3. Standard sections per subtype
    4. Onboarding template pattern (questions to ask if context missing)
    5. Distinction: context (mutable state) vs frameworks (immutable knowledge)
    6. Good vs bad examples
    7. Anti-patterns
  </action>
  <verify>Read file. Confirm: subtypes documented, standard sections defined, onboarding pattern shown, context vs frameworks distinction clear, good/bad examples, anti-patterns</verify>
  <done>AC-2 satisfied: Context spec covers subtypes, sections, onboarding, distinction from frameworks</done>
</task>
