<objective>
## Goal
Build the scaffolding workflow (`tasks/scaffold.md`) and authoring rules (`rules/*.md`) so that `/skillsmith scaffold` can consume a completed skill spec and generate a fully compliant skill directory with all files following syntax specs.

## Purpose
This is the second half of the meta-skill — discovery captures intent, scaffolding materializes it. Without this, users still have to manually create files and guess at syntax. With it, skill creation is end-to-end automated from interview to working directory.

## Output
- `skillsmith/tasks/scaffold.md` — Task workflow that reads a skill spec and generates all files
- `skillsmith/rules/*.md` — Authoring rules for each folder type (6 rules files, one per folder type that individual skills use)
</objective>

## AC-2: Entry Point Generated Correctly
```gherkin
Given a skill spec with identity, persona, activation, commands, and content architecture
When scaffolding generates the entry point
Then the file has correct YAML frontmatter (name, type, version, category, description, allowed-tools)
And contains all 5 XML sections (activation, persona, commands, routing, greeting)

```

## AC-3: Auxiliary Folders Generated Per Spec
```gherkin
Given a skill spec with content architecture listing tasks, frameworks, templates, context, and checklists
When scaffolding generates auxiliary files
Then only folders with entries in the spec are created (no empty folders)
And each file follows its respective syntax spec (correct sections, naming, structure)
And files contain meaningful scaffolded content (not just empty placeholders)
```

## AC-4: Rules Files Enforce Syntax Standards
```gherkin
Given the Skillsmith meta-skill
When rules files are loaded
Then each rule file has correct paths: frontmatter targeting its folder type
And contains must-have sections, naming conventions, and anti-patterns from specs
And rules are actionable (opencode can validate a file against them)
```

<task type="auto">
  <name>Task 1: Build scaffold workflow</name>
  <files>skillsmith/tasks/scaffold.md</files>
  <action>
    Create the scaffolding task following specs/tasks.md syntax. The workflow must:

    1. Purpose: Generate a compliant skill directory from a completed skill spec
    2. User story: Skill builder wants automated directory generation from their spec
    3. Steps (in order):

    **Step 1: locate_spec** (priority="first")
    - Ask user for the skill spec (they can paste it or reference a file path)
    - If the previous discover session just completed, offer to use its output
    - Parse the spec to extract: identity, persona, activation, commands, content architecture

    **Step 2: create_directory_structure**
    - Create `{skill-name}/` directory
    - Create subdirectories only for folder types that have entries in the spec
    - Follow kebab-case naming for all directories and files

    **Step 3: generate_entry_point**
    - Generate `{skill-name}/{skill-name}.md` following specs/entry-point.md:
      - YAML frontmatter: name, type, version (0.1.0), category, description, allowed-tools
      - `<activation>`: What, When to Use, Not For — from spec
      - `<persona>`: Role, Style, Expertise — from spec
      - `<commands>`: Command table — from spec (omit for task-only)
      - `<routing>`: Always Load (context files), Load on Command (tasks), Load on Demand (frameworks) — derived from content architecture
      - `<greeting>`: Brief intro + available actions + prompt — composed from skill identity

    **Step 4: generate_auxiliary_files**
    - For each folder type in the spec's content architecture:
      - **Tasks**: Generate file per specs/tasks.md — purpose, user-story, when-to-use, steps (with placeholder step names), output, acceptance-criteria. Steps should have meaningful names and structure based on the task's purpose, not generic placeholders.
      - **Frameworks**: Generate file per specs/frameworks.md — purpose section, core concepts skeleton, examples placeholder, anti-patterns placeholder
      - **Templates**: Generate file per specs/templates.md — header, intro, fenced template block with placeholder fields matching the skill's domain, field documentation
      - **Context**: Generate file per specs/context.md — header, standard sections based on subtype (user-profile, business-profile, or custom)
      - **Checklists**: Generate file per specs/checklists.md — header, purpose, categorized checklist items relevant to the skill's domain
    - Each generated file must contain enough scaffolded content to be immediately useful — NOT empty shells

    **Step 5: validate_and_report**
    - Load rules files (from rules/) and validate each generated file against its rule
    - Report: files created, any validation issues, next steps
    - Prompt: "Skill `{skill-name}` scaffolded. Review the files and customize them for your domain."

    4. Output: Complete skill directory with all files
    5. Acceptance criteria: Directory created, all files valid, user informed

    **What to avoid and WHY:**
    - Do NOT generate empty placeholder files ("TODO: fill in") — defeats the purpose of scaffolding
    - Do NOT create folders for folder types not in the spec — unnecessary clutter
    - Do NOT put process logic in the generated entry point — entry points are thin (route to tasks)
    - Do NOT use BDD acceptance criteria in generated task files — skills use plain checklists (per decision in Phase 1)
  </action>
  <verify>
    - scaffold.md has all required XML sections per specs/tasks.md
    - Step names are snake_case
    - Each step has specific instructions (not vague)
    - Output section specifies what gets created and where
    - Acceptance criteria are plain checklists
  </verify>
  <done>AC-1, AC-2, AC-3, AC-5 satisfied: Scaffold workflow can consume a spec and generate compliant files</done>
</task>

<task type="auto">
  <name>Task 2: Create authoring rules for each folder type</name>
  <files>skillsmith/rules/entry-point-rules.md, skillsmith/rules/tasks-rules.md, skillsmith/rules/templates-rules.md, skillsmith/rules/frameworks-rules.md, skillsmith/rules/context-rules.md, skillsmith/rules/checklists-rules.md</files>
  <action>
    Create 6 rules files in `skillsmith/rules/` following specs/rules.md syntax. Each rule file:

    1. Has YAML frontmatter with `paths:` targeting the correct glob pattern
    2. Contains "Must Have" sections extracted from the corresponding spec
    3. Contains "Naming Conventions" from the spec
    4. Contains "Anti-Patterns" from the spec
    5. Is concise and actionable — opencode can validate a file against these rules

    **Specific rules per file:**

    **entry-point-rules.md** (paths: `skills/*/*.md` excluding tasks/frameworks/etc)
    - Must have: YAML frontmatter (name, type, version, category, description), all 5 XML sections
    - Naming: kebab-case directory, entry point matches directory name
    - Anti-patterns: process logic in entry point, generic XML tags, missing activation boundaries

    **tasks-rules.md** (paths: `skills/*/tasks/*.md`)
    - Must have: purpose, user-story, when-to-use, steps, output, acceptance-criteria
    - No frontmatter
    - Steps must have name attribute (snake_case)
    - Anti-patterns: unnamed steps, vague instructions, BDD criteria, monolithic tasks

    **templates-rules.md** (paths: `skills/*/templates/*.md`)
    - Must have: header, intro with file naming pattern, fenced template block, field documentation
    - Placeholder conventions: {curly} = variable, [square] = prose
    - Anti-patterns: template without field docs, mixing placeholder types

    **frameworks-rules.md** (paths: `skills/*/frameworks/*.md`)
    - Must have: purpose section, core concepts, examples
    - No frontmatter (read-only knowledge)
    - Anti-patterns: prescriptive instructions (frameworks teach, they don't command)

    **context-rules.md** (paths: `skills/*/context/*.md`)
    - Must have: header, standard sections based on subtype
    - Context is the ONLY mutable file type — document this
    - Anti-patterns: storing process state, config that belongs in frontmatter

    **checklists-rules.md** (paths: `skills/*/checklists/*.md`)
    - Must have: header, purpose, categorized items with `- [ ]` format
    - Each item must be independently pass/fail
    - Anti-patterns: vague criteria, mixing instructions with validation

    **What to avoid and WHY:**
    - Do NOT duplicate the full specs into rules — rules are compact enforcement, specs are comprehensive reference
    - Do NOT create a rules file for the rules folder itself — that's meta-recursion with no value
  </action>
  <verify>
    - Each rules file has YAML frontmatter with paths: field
    - Each rules file has Must Have, Naming, and Anti-Patterns sections
    - Rules are concise (not copies of specs)
    - No rules file for the rules folder itself
  </verify>
  <done>AC-4 satisfied: Rules files enforce syntax standards for all 6 folder types</done>
