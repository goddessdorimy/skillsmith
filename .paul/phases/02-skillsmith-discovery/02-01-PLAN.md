<objective>
## Goal
Build the Skillsmith meta-skill entry point and discovery workflow — the conversational interview that captures everything needed to scaffold a new skill.

## Purpose
Discovery is the front door of the framework. Without it, users must manually assemble skill files from specs. With it, a guided conversation produces a complete skill spec that the scaffolding engine (Phase 3) can consume.

## Output
- `skillsmith/skillsmith.md` — Meta-skill entry point (type: suite)
- `skillsmith/tasks/discover.md` — Discovery workflow (interview → skill spec)
- `skillsmith/templates/skill-spec.md` — Output template for discovery results
</objective>

<task type="auto">
  <name>Task 2: Create discovery workflow</name>
  <files>skillsmith/tasks/discover.md</files>
  <action>
    Create the discovery interview task following specs/tasks.md:

    **No frontmatter** (task files don't use it per spec).

    **XML Sections:**

    `<purpose>`: Guided conversational interview that captures all decisions needed to scaffold a skill.

    `<user-story>`: As a skill builder, I want a structured interview that walks me through every decision so that I get a complete skill spec without missing anything.

    `<when-to-use>`: Creating a new skill from scratch. Also useful for documenting an existing skill's design.

    `<context>`: @specs/entry-point.md (for frontmatter fields and section structure)

    `<references>`: @specs/*.md (lazy-load individual specs as each topic comes up)

    `<steps>`: Design interview as grouped phases, NOT one giant checklist. Each phase is a natural conversation chunk:

    **Phase 1 — Identity** (name, type, category, description):
    - Ask skill name (enforce kebab-case)
    - Ask type: suite / standalone / task-only — explain tradeoffs briefly
    - Ask category (suggest common: operations, content, development, design, analysis)
    - Ask one-line description

    **Phase 2 — Persona** (role, style, expertise):
    - Ask: who does this skill act as?
    - Ask: how does it communicate? (structured/casual/opinionated/etc.)
    - Ask: what domains does it know deeply?

    **Phase 3 — Scope & Commands** (activation, commands):
    - Ask: what does this skill do? (What section)
    - Ask: when should someone reach for it? (When to Use)
    - Ask: what is explicitly NOT this skill's job? (Not For)
    - If suite/standalone: ask what commands/sub-commands it offers

    **Phase 4 — Content Architecture** (routing, folders needed):
    - Walk through each auxiliary folder type:
      - Tasks needed? (what workflows)
      - Frameworks needed? (what domain knowledge)
      - Templates needed? (what output structures)
      - Context needed? (what user/business state)
      - Checklists needed? (what quality gates)
    - For each "yes", capture: file name, brief purpose, loading condition (always/on-command/on-demand)

    **Phase 5 — Review & Confirm**:
    - Display complete skill spec using templates/skill-spec.md format
    - Ask for confirmation or edits
    - On confirm: output is the filled spec, ready for scaffold phase

    **Step behavior:**
    - Use `<step>` tags with `name` attributes per spec
    - Each step waits for user response before proceeding (conversational, not dump-and-run)
    - Validate inputs inline (e.g., kebab-case check on name)

    `<output>`: Filled skill-spec template written to `.paul/phases/current/skill-spec-{name}.md` or presented inline

    `<acceptance-criteria>`: Plain checkbox list — all 5 phases completed, spec output generated, all fields populated
  </action>
